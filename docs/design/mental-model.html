§ 0 ─ framing

"满足各种爱好" 与 "限死" 是假二选一。

通用工具都走第三条路 · strong invariants + weak style

你纠结的二选一(对齐 git 的爱好 vs 限死)是平行视角的对立。 所有"通行 + 大家都舒服"的产品都不是其中之一——它们都精挑出 一组真不能妥协的事强制锁住,其他完全放开。

git 自己的问题恰恰是 invariants 太薄——只锁 hash chain、content addressable、DAG,所有"协作形态"全让用户拼。所以才有 GitHub Flow / TBD / git-flow / stacked diffs 几大流派打架,AI 在中间漂移。

alt 的灵魂位置:
底层强 invariants(签名、op log、portable patch、workspace、git 兼容、immutable history)
中间 flow 是可换 preset
顶层 surface 按"用户当下的需求维度"暴露概念,不按用户类型分

这样既不像 git 那样让用户自己拼到漂移,也不像我 v1 提的 task model 把所有人锁死——alt 像 Postgres 一样硬 invariants、像 ESLint 一样可换 preset、像 vim 一样可适配多种风格。

另外两条架构原则同样不可妥协: alt ⊇ git 三轴(format-native § 6 · visibility 场景 84 · structured commit msg 场景 85 — alt 仓比 git mirror 多三层信息) + LLM 隔离(场景 86 · core 0 LLM · 增强 well-defined · invariant 7 候选)。

prior art: 这种 "底强、上活" 的分层不是我创新——是 所有通用工具的共同形态。alt 真要创新的位置不在分层本身,而在 invariants 的具体选择。

Linux kernel VFS / syscall 契约 / 进程模型 user-space 怎么写
Postgres ACID / SQL 标准 / MVCC schema / index 类型 / 隔离级别

§ 1 ─ invariants

底层 6 件事 — 永远成立。

alt 的灵魂在这里 · 切 preset 不动它们

下面 6 条是 alt 比 git 多锁的东西。git 只锁 3 件(hash chain、content addressable、DAG),alt 再加 6 件——这 9 件加起来,是无论用户走哪 种 preset 都不会塌方的底。

这 6 件每一件都是设计 bet,但每一件都有 prior art 撑腰——不是 凭空发明,而是把别的 VCS 验证过的好做法都强制成默认。

i.

Signed Op

每一次状态变更都签名归属。AI / human / fork 都是 first-class actor,签名身份强制 attach。

why"谁做了这个改动" 永远可问。AI-coding 时代追责必须可信。

ii.

Op Log

append-only 的操作历史,不只是 undo 工具——是 navigation primitive。可以"回到 3 小时前的状态"。

priorjujutsu 的 op log。alt 把它做成一等公民 UX,不藏在专家层。

iii.

Portable Patch

任何 change 都能序列化成内容寻址、签名的便携 patch。pull / send / sneakernet 都是同一格式。

priorgit format-patch + Pijul patch theory。alt 让它成 protocol-level 一等公民。

iv.

Workspace

每个 actor 一个隔离沙盒,并发安全。多个 workspace 可在飞, 互不踩。

priorPlastic SCM + git worktree + jj working copy。alt M5 已落地。

v.

Git Compat

byte-exact git 兼容。alt 仓和 git 仓可以无损往返。任何 git 工具都能读 alt 产出的 history。

whyalt 的 north star。 没这个 alt 就是另一个孤立 VCS,不能跟 GitHub / IDE 共生。

vi.

Immutable History

commit 一旦写下不可消失。amend / rebase 创建新 commit,旧 的留在 op log 里可追溯。

why跟 git 一致。 AI 操作不允许"消失",任何动作都可审计。

§ 2 ─ preset

中层 flow preset — 像 ESLint 的 config。

user 选口味 · 底层 invariants 不变

preset 不是 "mode",不是把 alt 配成另一个工具。它是装在 invariants 之上的一组 validator + lifecycle 命令,告诉 alt: "我用 GitHub Flow / TBD / AVH 的规矩工作,请按这些规矩 lint 我"。

preset 之间切换不影响 commit 历史、不影响签名、不影响 op log——只 影响:用什么命令、什么时候报警、reviewer 怎么派、release 怎么 cut。

第一批 ship 4 个 + 1 个 custom slot。后续 preset 是社区可扩展点, 类似 ESLint 的 shareable config。

§ 3 ─ protocol

不管哪个 preset, change 都走同一条 protocol。

produce · sign · review · land

review / land / patch 是 protocol-level,不是 workflow-level。任何 change 不管在哪个 preset 下,都按 produce → sign → review → land 四步流转。preset 提供的只是 transport(怎么传、怎么呈现、 怎么 gate)。

这是 alt 在底层 invariants 之上加的一层"协议骨架":AI 不需要 理解每个 preset 的 quirk,只需要按 protocol 行事——底层 protocol 不变,上层 preset 把它投影成 PR / Gerrit Change / stack push / mailing-list patch / AVH feature-finish 等不同 transport。

这一层是真创新。把"如何把一个 change 送到 main" 抽象成单一 protocol,跨 preset 复用,是其他 VCS 都没做的。

图 3 · 协议层

无论 preset 是 PR / stack / AVH finish / mailing-list,所有 change 都走 produce → sign → review → land 同一 protocol。preset 只是 transport 的差异。

§ 4 ─ dimensions

4 个需求维度,不是 4 类用户。

D1 写代码 · D2 管 change · D3 协作 · D4 治理

修正 v2 的早期错误:不能假设"单人 = 简单 surface、团队 = 复杂 surface"。一个 AI agent 单干很可能比 10 人团队更复杂——它要 review 自己历史、undo 错误、跨多 change 并行、解 op log。actor 数 量不等于心智复杂度。

正确的轴是 当下的工作类型:你在写代码?管理一个 change 的生命周期?跟别人协作?治理 / 审计? 不同维度暴露不同概念。同一个 user(包括 AI、包括单人、包括团队) 会在同一天里跨好几个维度。

4 个维度按"心智负担递增"叠加,但顺序不绑死用户类型:

D1 · 写代码——commit / branch / remote。 git 用户已熟。任何人起步。
D2 · 管 change——change(atomic 单位) / op log / undo。jj 风格的精细化。
D3 · 协作——preset / workspace / review。 选口味 + 多 actor 同 repo。
D4 · 治理——policy / codeowners / audit。 合规、capability 强制。

每个 D 是同一个 user 按需触达,不是不同 user 看到不同 menu。AI agent 一上来就可能跨 D1+D2+D3——这是常态,不是异常。

关键区别 "概念暴露"是按需驱动的,不是 alt 强制递进。用户做什么触达 哪个 D:
  • "我刚刚 commit 错了" → 触达 D2 的 undo 和 op log
  • "我要跟 alice 一起做这个" → 触达 D3 的 workspace 和 review
  • "这次改动碰了敏感路径" → 触达 D4 的 codeowners
不是 alt 隐藏命令,是 user 在不碰到那个场景时不需要那个概念。

图 4 · 4 维度

4 个维度按"心智负担递增"排,但不绑用户类型。AI / solo / 团队成员都按当下需求触达对应 D,双向自由跨。

scenario 1 · v3 格式示范

新项目 · 从零起一个 alt 仓 🟡 partial

创建仓

▼ 你要打的
alt用户输入
$ mkdir myproj $ cd myproj $ alt init Initialized empty alt repository in ./.alt
git用户输入
$ mkdir myproj $ cd myproj $ git init Initialized empty Git repository in ./.git/
alt 静默替你做了 除了创建 .alt/(对应 .git/),还自动:
  • 初始化 op log(空)
  • 创建默认 workspace 名叫 default
  • 设 HEAD = unborn main
  • 允许 unborn main 上第一次 commit(invariant 6 例外——没 flow init 之前)

设身份(首次用 alt 才需要 · 全局一次)

▼ 你要打的
alt用户输入
$ alt identity add you@example.com → ed25519 key pair generated → stored in ~/.alt/identity/you@example.com/
git用户输入(分两条)
$ git config --global user.name "you" $ git config --global user.email you@example.com
alt 静默替你做了
  • 生成 ed25519 密钥对(sec/pub),sec 加密存盘
  • 注册到全局 identity store
  • 所有 alt 仓共享身份——这个 identity 是跨仓全局的,不是 per-repo
深一层 git 的 user.name / user.email 是文本字段,任何人改成 你的邮箱也行——没有密码学保证。alt 用真密钥对,commit signature 不可伪造(invariant 1)。 未来 alt identity use <key> 支持多身份切换(场景 25),但今天没 ship。

▼ 你想打但今天不能

$ alt preset use github-flow → preset = github-flow · written to .alt/config → trunk = main · topic branches without prefix → review-then-land enforced

今天 alt 静默替你做了

  • 所以这步今天可跳过——你需要 avh-flow 才需要显式切

首次 commit

▼ 你要打的
alt用户输入
$ echo "# myproj" > README.md $ alt add README.md $ alt commit -m "init" [main (root-commit)] a1b2c3d4 → signed: you@example.com
git用户输入
$ echo "# myproj" > README.md $ git add README.md $ git commit -m "init" [main (root-commit) a1b2c3d] init
alt 静默替你做了
  • commit object byte-exact 跟 git 兼容(invariant 5)——任何 git 工具都能读
  • 自动用你的 ed25519 sec 签名,signature 嵌进 commit header
  • 记 op log:_op_001 commit · main · a1b2c3d
  • git 读这个 commit 时 signature header 当 unknown 跳过,正常 work

加 GitHub remote + 首次 push

▼ 你要打的
alt用户输入
$ alt remote add origin https://github.com/you/myproj.git $ alt push -u origin main push 'origin': 4 object(s), 318 byte(s) ok refs/heads/main
git用户输入
$ git remote add origin https://github.com/you/myproj.git $ git push -u origin main To github.com:you/myproj.git * [new branch] main -> main
alt 静默替你做了 — 三处自动
  • ① auth:先看环境变量 ALT_HTTP_USER_ORIGIN / ALT_HTTP_TOKEN_ORIGIN(remote 名大写);没设就 fallback 跑 git credential fill(走你的 gh auth token / keychain)。 token 永远不进 remote URL。
  • ② SSH→HTTPS rewrite:如果你给 SSH 形式 URL (git@github.com:you/myproj.git),alt 自动转 HTTPS——5 个 well-known hosts 内置(github / gitlab / bitbucket / codeberg / sr.ht)。其他主机用 [url "X"] insteadOf = Y 配置。
  • ③ transport:alt 自家 git-smart-http v2 实现, 不需要本地装 git 也能 push 到 GitHub。
git 的对应行为 git push 默认是 credential.helper(系统 keychain 或 osxkeychain), 首次 auth 弹窗 + 存进 keychain。alt 多了 env-vars 优先级 + 不存 token 进 URL + SSH→HTTPS 自动转,适合 AI agent 用(env 注入比 keychain 容易)。

§ 6 ─ format-native

每种格式自己的 history / diff / edit。

owner 战略重点 · alt 跟 git 最大差异 · git 历史包袱补不了

owner 原话:"除了 alt 的技术基础架构, 还有下决心为每种格式独立做 history / diff / edit, 这个流程优化就是我们最重要的事了, 而且也是 git 最难做的事 因为有历史包袱"

git 把所有 binary 当 opaque blob——diff 输出 "Binary files differ" 一句话, review 没法做,blame 不能跟,merge 必然冲突。每种格式各有结构:json 的 key 顺序无关、 xlsx 的 sheet/row/cell 三层、png 的像素区域、pdf 的页面 + 元数据、safetensors 的 tensor + shape——每种格式都有 "结构化 diff" 的可能,但 git 没设计这层。

alt 用 prism 抽象:每种格式有自己的 prism, 决定如何 chunk(存储优化)、diff(语义对比)、render(review 展示)、 edit(in-place 修改)。已 ship 的 4 个 prism (deflate / png / zip / store) 是地基,真正的格式 UX 大量在 M16 后续段——下面把每种格式的完整流程链摊开看。

每个格式场景看 4 件事: 1. 这格式 today git 怎么糟 · 2. alt 该怎么做(目标体验) · 3. alt 今天哪个 prism / 命令 ship · 4. 缺哪些。

§ 7 ─ AI 时代特有场景

AI agent 工作流 · alt-first。

git 完全没设计 · alt 的 AI-first 主战场

AI agent 跟 human 工作流的差异不止"机器跑得快"。是结构性不同: (a) agent 一次会 commit 几十次(human 一周几十次); (b) agent 会全错(整 session 方向偏),需要 batch rollback; (c) agent 跑得没意识,要 cap-scope 在 commit 之前拦截; (d) agent 之间会协作(handoff 任务),需要 attribution + identity 切换可信。

git 完全没设计这层。alt 的 invariants (op log + signed op + workspace + capability) 是 地基,下面 6 个场景是真要落地的 AI 时代工作流。

§ 9 ─ Web UI

§ 12 ─ anti-rot · 元层

所有 reference 都是 typed pointer · system 持续 verify · drift 立刻 catch。

invariants 的免疫系统 · alt ⊇ git 第四轴候选

owner 原话:"加一个核心话题,anti-rat 或者说 single source multi pointers,结构化关联,自查自纠。"

这一节是 alt://principle/anti-rot 的入口 · 完整架构决策见 docs/design/anti-rot-protocol.md ADR。

原子层(atom):alt 内每个 typed 概念 (scenario / invariant / cmd / prism / commit / path / issue …) 都是一个 atom。每个 atom 有三层身份:semantic kebab(URL 槽 · 人类用)+ uuid v7(internal stable · rename 时 ref 不破)+ content_hash(verification · 跟 alt://invariant/immutable 一体)。

typed URL:alt://<kind>/<id>[@pin][/sub]。 这条文档以前写 "见场景 17 amend"——free text · gate 1 audit 每改场景号都漏 update。 转 typed 后写 alt://scenario/amend · 改号自动 alias chain redirect · 再不漏。

anti-rot 是 meta · 不是 invariant 8。
它不锁 state · 它锁 reference structure。

  2. LLM 隔离原则(场景 86 · invariant 7 候选)—— "core 0 LLM · 增强 well-defined · 不允许乱来"。 架构边界让 alt 从 M18 起就不会失控地引入 LLM 依赖。 类比 Postgres core ↔ pgvector / VS Code core ↔ Copilot。 AI 时代真正可信的 VCS 必须能"完全离线照常跑 + 选择性增强"。
